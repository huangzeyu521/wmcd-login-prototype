## 一、技术栈最终选型

### 1.1 选型原则
1. **可预览优先**：原型需可直接在浏览器打开（无需构建），便于Artifacts/文档级预览
2. **国资合规**：避免重度依赖境外云服务SDK；支持后续平滑迁移至Vue/React企业级框架
3. **开发效率**：Tailwind CDN实现快速样式；ECharts强数据可视化；原生JS避免构建复杂度
4. **平滑升级**：HTML原型结构可直接映射为Vue/React组件，便于后期迁移

### 1.2 最终选型
| 层级 | 技术 | 版本 | 理由 |
|------|------|------|------|
| 标记语言 | HTML5 | - | 标准 |
| 样式框架 | **Tailwind CSS**（CDN） | 3.x | 原子化CSS、开发速度快、一致性好、易于维护 |
| 自定义样式 | 内联 CSS + 共享 shared.css | - | WMCD设计系统变量、覆盖默认样式 |
| JS 运行时 | 原生 JavaScript (ES6+) | - | 无需构建工具，浏览器直接运行 |
| 数据可视化 | **ECharts** | 5.4.3 | 功能最强大的国产图表库，政企广泛采用 |
| 图标 | Heroicons (内联SVG) + Emoji | - | MVP阶段快速实现，后期替换SVG套装 |
| 字体 | 系统字体 + Google Fonts 备用 | - | 符合国资网络环境 |
| HTTP 预览 | Python `http.server` | 3.x | 0依赖起静态服务器 |

### 1.3 后期生产环境升级路径（非本期交付）
> 本期交付为静态HTML原型，后期可平滑迁移至企业级栈：
- **框架**：Vue 3 + TypeScript 5 + Vite 5
- **UI 库**：Ant Design Vue 4.x / Element Plus
- **状态管理**：Pinia + TanStack Query
- **路由**：Vue Router 4
- **构建部署**：Vite + Nginx + CDN
- **迁移工作量**：预估 2-3 周可完成（结构与样式已确立）

## 五、代码规范

### 5.1 HTML 规范
- 文档类型：`<!DOCTYPE html>`
- 语言：`<html lang="zh-CN">`
- 字符编码：UTF-8
- 视口：`<meta name="viewport" content="width=1280">`（PC大屏优先）
- 语义化：合理使用 `<header> <nav> <main> <section> <aside> <footer>`
- 可访问性：所有图片含 `alt`，按钮含 `aria-label`（重要按钮），输入有 `label`

### 5.2 CSS 规范
- **命名约定**：BEM 变体 `wm-{模块}-{元素}-{修饰}`（如 `.wm-card`, `.wm-btn-primary`）
- **工具类**：Tailwind CDN（多用 flex、grid、间距、字号）
- **自定义变量**：统一在 `:root` 声明（见 `shared.css`）
- **禁止**：`!important`（特殊场景必须加注释说明）
- **模块化**：每个 HTML 内置的 `<style>` 仅覆盖页面专属样式，公共样式放 `shared.css`

### 5.3 JavaScript 规范
- **ES6+**：`const`/`let`、箭头函数、模板字符串、解构
- **命名**：`camelCase` 变量函数、`UPPER_SNAKE_CASE` 常量
- **IIFE封装**：每段图表初始化用 `(function(){ ... })()` 包裹，避免全局污染
- **ECharts实例**：`resize` 事件统一监听
- **工具函数**：统一放 `shared.js`

### 5.4 数据模拟约定
本期所有数据为前端静态Mock：
- 数字拟真（规模、单位符合国资语境）
- 企业名称使用"皖能/海螺/建工/皖企投"等真实安徽省属企业名称
- 时间戳保持自洽
- 图表数据具备趋势合理性（不做无逻辑的随机噪声）

### 5.5 注释规范
```html
<!-- ===== 模块名称 ===== -->
<!-- 描述：xxx -->
```
```js
// ===== 模块名称 =====
// 描述：xxx
```

## 六、Git 提交规范（后期版本控制）

### 6.1 分支策略
- `main`：主干，仅接受合并的稳定版本
- `develop`：日常开发分支
- `feature/agent-xxx`：功能开发
- `fix/xxx`：缺陷修复
- `release/vX.Y.Z`：版本发布

### 6.2 Commit Message 格式
```
<类型>(<范围>): <简短描述>

<详细描述>（可选）
```

| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | 修复缺陷 |
| style | 仅样式调整 |
| refactor | 重构，无功能变化 |
| docs | 文档变更 |
| chore | 构建/工具链 |
| perf | 性能优化 |
| test | 测试相关 |

示例：
```
feat(agent-credit): 增加企业信用总览卡六维雷达图
fix(shared): 修复 Topbar 在1280以下宽度溢出问题
style(dashboard): 统一所有卡片阴影与圆角
```

### 6.3 PR 规则
- 必须关联需求或缺陷编号
- 必须经过 1+ 人 Review
- 必须通过本地预览验证
- 合入前必须解决所有冲突

## 九、风险与应对

| 风险 | 可能性 | 影响 | 应对 |

| CDN不可访问 | 低 | 中 | 提供离线包fallback（`assets/lib/`） |

| 数据可视化性能 | 低 | 低 | ECharts已优化，本期数据量小 |
| 后期迁移到Vue/React工作量大 | 中 | 中 | 结构清晰、样式分离、易迁移 |
